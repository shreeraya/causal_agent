The problem: supply chains can answer what, but not why

Supply chain teams ask two fundamentally different kinds of questions — and today's tooling only handles one of them.

Factual questions

"How much inventory of SKU-0042 is at the Northeast RDC?" · "Which supplier provides our packaging film?" · "What's the lead time from GlobalSweet to Boston?" — answerable by lookups in a database or dashboard.

Causal questions

"Why did stockouts spike in western stores?" · "What was the effect of the promotion on sales?" · "Which SKUs are at risk if a supplier fails?" — these need structural knowledge and counterfactual reasoning, not lookups.

Why existing tools fall short

  • Dashboards & BI show correlations in aggregates. They can show stockouts rose, but not whether a port delay, a promotion, or seasonality caused it.
  • Relational databases struggle with the question "what does this event touch?" — tracing a supplier failure through bills of materials to at-risk store-SKUs is a multi-hop path query, which is exactly what graphs are for.
  • LLMs alone speak the business language but hallucinate numbers and have no access to your network structure or history.

The hypothesis this demo tests

Combining four ingredients yields a system that answers both question classes with evidence:

  1. A knowledge graph (Neo4j) that stores the supply chain's structure — who supplies what, what's made from what, what ships where — so causal pathways are first-class, traversable data.
  2. An ontology — a formal, shared vocabulary of entity types and relationships — that lets a language model translate business questions into precise graph queries without guessing.
  3. Time-series facts (weekly snapshots) that provide the before/after variation causal inference needs.
  4. An LLM agent that classifies the question, plans queries, and explains the answer — grounded so every number comes from the graph.
Why synthetic data? The dataset is a simulated consumer-goods network where the true causal effects are known by construction — promotions lift demand ×1.4, disruptions cut supply with a one-week lag per echelon. That means the agent's causal answers can be scored against ground truth, which is impossible with real-world data. The methodology transfers unchanged to real data.

Architecture

A thin, transparent stack: every answer is produced by live Cypher queries against the graph — the LLM never answers from memory.

The agentic loop, step by step

  1. Your question (plus conversation history) goes to the LLM with a system prompt containing the ontology, an entity vocabulary queried from the graph at startup, and canonical causal query patterns.
  2. The model classifies the question — factual or causal — and issues one or more Cypher queries through its single tool, run_cypher.
  3. Each query is screened (write clauses rejected, executed inside a read-only transaction, results capped at 60 rows) and the rows return to the model as JSON.
  4. The model iterates — refining queries, recovering from errors — until it can compose an answer from the query results only.
  5. The UI renders the answer plus the full query trace, so you can audit exactly what evidence produced it.

Safety properties

  • The database is opened read-only to the agent: write clauses are rejected by pattern and queries run in read transactions.
  • API keys (OpenAI, Neo4j) live server-side only; the browser never sees them.
  • Access code + per-IP rate limiting + message-size caps protect the public demo endpoint.

How the ontology is used at runtime

  • The full schema above is embedded in the agent's system prompt — so it writes correct Cypher patterns instead of guessing structure.
  • An entity vocabulary (every supplier, material, plant, DC, city, and event name) is queried from the graph when the server starts. Graph string-matching is case-sensitive; the vocabulary means the model copies exact names instead of inventing them.
  • The semantics ("MADE_FROM is the BOM", "supply disruptions propagate downstream with ~1 week lag per echelon") turn schema into causal knowledge the model can reason with.
The graph was seeded by deterministic Python scripts (same random seed → same graph), then the 52-week history was produced by a small multi-echelon simulation — order-up-to inventory policies, seasonality, promotions, and disruption shocks that reduce inbound supply along affected lanes. Everything runs on a free Neo4j Aura cloud instance.

Causal + Neo4j + ontology: how the three integrate

Each ingredient answers a question the other two cannot. Together they turn "why?" into a query plan.

1 · Ontology — the language

Maps business terms to graph structure: "sugar shortage" → a DisruptionEvent; "exposed SKUs" → Products reachable via SUPPLIES + MADE_FROM. Without it, the LLM guesses schema and hallucinates entities.

2 · Neo4j graph — the pathways

Causal influence travels along edges: event → supplier → material → BOM → SKU → DC → store. Multi-hop traversal finds the treated group (exposed SKUs/stores) and the control group (everything else) — the hard part of causal analysis.

3 · Causal inference — the evidence

Weekly snapshots supply before/after variation. Difference-in-differences over treated vs control × before vs during separates the event's effect from seasonality and noise.

The pipeline for a causal question

  1. Classify — is this factual (lookup) or causal (effect/attribution)? Factual questions go straight to Cypher.
  2. Trace the pathway — traverse the graph from the treatment to its exposure set, e.g. (event)-[:AFFECTS]->(supplier)-[:SUPPLIES]->(material)<-[:MADE_FROM]-(sku).
  3. Build cohorts and windows — treated vs control SKUs/stores; before vs during/after weeks, shifted for propagation lag (~1 week per echelon).
  4. Aggregate the four cells — stockout rate, units received, units sold, computed in Cypher over ~100k snapshots.
  5. Report effect size + caveats — differences and ratios, an explicit note on possible confounders, and honesty about correlation vs causation.

Worked example: "Did the sugar shortage cause stockouts?"

The event hits one supplier in April 2026. The graph traversal finds the exposed SKUs; the snapshots give a natural difference-in-differences:

Store-level stockout rateBefore impact windowDuring impact windowChange
Exposed SKUs (treated)~0.9%~1.8%roughly doubles
Unexposed SKUs (control)~0.4%~0.4%flat

Controls staying flat in the very same weeks rules out seasonality and store-wide shocks — the parallel-trends logic of diff-in-diff. The effect also appears ~5 weeks after the event start, matching the expected propagation lag through plant → central DC → regional DC → store.

Honest limitations

  • This is observational causal inference — diff-in-diff with graph-derived cohorts — not a randomized experiment. Trap questions (e.g. "did promotions cause the December snack spike?" — it's seasonality) test whether the agent respects that line.
  • Effect estimates are aggregate comparisons; a production system would add a dedicated causal engine (DoWhy / EconML) for regression adjustment, placebo tests, and sensitivity analysis, with the graph still providing cohorts and pathways.
  • The LLM plans queries; small models occasionally under-query. The visible query trace keeps every answer auditable.
