The problem: supply chains can answer what, but not why

Supply chain teams ask two fundamentally different kinds of questions — and today's tooling only handles one of them.

Factual questions

"How much inventory of SKU-0042 is at the Northeast RDC?" · "Which supplier provides our packaging film?" · "What's the lead time from GlobalSweet to the Northeast RDC?" — answerable by lookups in a database or dashboard.

Causal questions

"Why did stockouts spike in the western network?" · "What was the effect of the promotion on sales?" · "Will SKU-0070 stock out next month — and why?" · "Would that stockout have happened if the supplier fire hadn't occurred?" — these need structural knowledge, projection, and counterfactual reasoning, not lookups.

Why existing tools fall short

  • Dashboards & BI show correlations in aggregates. They can show stockouts rose, but not whether a port delay, a promotion, or seasonality caused it.
  • Relational databases struggle with the question "what does this event touch?" — tracing a supplier failure through bills of materials to at-risk DC-SKUs is a multi-hop path query, which is exactly what graphs are for.
  • LLMs alone speak the business language but hallucinate numbers and have no access to your network structure or history.

The hypothesis this demo tests

Combining five ingredients yields a system that answers all of these question classes with evidence:

  1. A knowledge graph (Neo4j) that stores the supply chain's structure — who supplies what, what's made from what, what ships where — so causal pathways are first-class, traversable data.
  2. An ontology — a formal, shared vocabulary of entity types and relationships — that lets a language model translate business questions into precise graph queries without guessing.
  3. Time-series facts — 52 weeks of history plus a 25-week projected inventory plan (from demand forecasts and inventory-balance equations) so future stockout risk is queryable, not just past performance.
  4. A causal engine (DoWhy) — backdoor-adjusted effect estimation with refutation tests, what-if simulation, and true counterfactuals over a structural causal model fitted to the snapshots.
  5. An LLM agent that classifies the question, plans queries and causal analyses, and explains the answer — grounded so every number comes from a tool result.
Why synthetic data? The dataset is a simulated consumer-goods network where the true causal effects are known by construction — promotions lift demand ×1.4, disruptions cut supply with a one-week lag per echelon, three SKUs run ~25% above forecast. That means the agent's causal answers can be scored against ground truth, which is impossible with real-world data. The methodology transfers unchanged to real data.

Architecture

A thin, transparent stack: every answer is produced by live Cypher queries against the graph — the LLM never answers from memory.

The agentic loop, step by step

  1. Your question (plus conversation history) goes to the LLM with a system prompt containing the ontology, an entity vocabulary queried from the graph at startup, and a five-way question taxonomy with canonical patterns.
  2. The model classifies the question — factual, causal effect, what-if, counterfactual, or future risk — and picks tools: run_cypher for lookups and cohort discovery, estimate_effect / what_if / counterfactual / explain_stockout_risk for the causal work.
  3. Cypher is screened (write clauses rejected, read-only transactions, 60-row cap); causal tools run DoWhy over a pandas panel of all 63k snapshots with a structural causal model fitted at startup.
  4. The model iterates — refining queries, recovering from errors — until it can compose an answer from tool results only.
  5. The UI renders the answer plus the full tool trace, so you can audit exactly what evidence produced it.

Safety properties

  • The database is opened read-only to the agent: write clauses are rejected by pattern and queries run in read transactions.
  • API keys (OpenAI, Neo4j) live server-side only; the browser never sees them.
  • Access code + per-IP rate limiting + message-size caps protect the public demo endpoint.

How the ontology is used at runtime

  • The full schema above is embedded in the agent's system prompt — so it writes correct Cypher patterns instead of guessing structure.
  • An entity vocabulary (every supplier, material, plant, DC, and event name) is queried from the graph when the server starts. Graph string-matching is case-sensitive; the vocabulary means the model copies exact names instead of inventing them.
  • The semantics ("MADE_FROM is the BOM", "supply disruptions propagate downstream with ~1 week lag per echelon", "demand attainment above ~115% of forecast erodes inventory") turn schema into causal knowledge the model can reason with.
The graph was seeded by deterministic Python scripts (same random seed → same graph), then the 52-week history was produced by a small multi-echelon simulation — order-up-to inventory policies planned to forecast, seasonality, promotions, and disruption shocks that reduce inbound supply along affected lanes. The 25 projected weeks extend the same inventory-balance equations forward deterministically: projected demand = forecast × recent demand attainment, receipts capped by the committed supply plan. Everything runs on a free Neo4j Aura cloud instance.

Causal + Neo4j + ontology: how the three integrate

Each ingredient answers a question the other two cannot. Together they turn "why?" into a query plan.

1 · Ontology — the language

Maps business terms to graph structure: "sugar shortage" → a DisruptionEvent; "exposed SKUs" → Products reachable via SUPPLIES + MADE_FROM. Without it, the LLM guesses schema and hallucinates entities.

2 · Neo4j graph — the pathways

Causal influence travels along edges: event → supplier → material → BOM → SKU → DC → store. Multi-hop traversal finds the treated group (exposed SKUs/stores) and the control group (everything else) — the hard part of causal analysis.

3 · DoWhy engine — the evidence

A structural causal model fitted to the weekly snapshots. Effect questions get backdoor-adjusted estimates with placebo refutation tests; what-ifs get interventional simulation; "would it have happened?" gets true rung-3 counterfactuals.

The pipeline for a causal question

  1. Classify — factual, effect, what-if, counterfactual, or future risk.
  2. Trace the pathway — traverse the graph from the treatment to its exposure set, e.g. (event)-[:AFFECTS]->(supplier)-[:SUPPLIES]->(material)<-[:MADE_FROM]-(sku) — the graph builds the treated/control cohorts.
  3. Estimate — the DoWhy engine adjusts for confounders (seasonality via each SKU's seasonal baseline, promotions) and runs refutation tests; a quick Cypher diff-in-diff can cross-check it.
  4. Report effect size + caveats — with refutation status, and projected-week numbers clearly labelled as plan, not observation.

Why future stockouts are explainable

Every snapshot carries the demand forecast alongside actuals, and replenishment is planned to forecast with receipts capped near 115% of it. The 25 projected weeks extend the inventory balance forward (projected demand = forecast × recent demand attainment). So when a projected week stocks out, the cause is decomposable: demand running persistently above forecast, an ongoing or announced disruption cutting the lane's supply_factor, a depleted starting position, a planned promotion spike — or several at once. explain_stockout_risk names each driver with its evidence.

Honest limitations

  • This is observational causal inference on top of a fitted SCM — not a randomized experiment. Trap questions (e.g. "did promotions cause the December snack spike?" — it's seasonality) test whether the agent respects that line.
  • What-ifs and counterfactuals are single-week: inventory carryover between weeks is held at observed values. Multi-week interventions ("expedite 500 units in week 53") need a re-projection tool.
  • The 25-week projection is deterministic — one path, not a distribution. A Monte Carlo layer would turn risk flags into probabilities.
  • The LLM plans tool calls; small models occasionally under-query. The visible tool trace keeps every answer auditable.
